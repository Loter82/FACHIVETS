# Unipro Loyalty Research — Fachivets (v2, corrected)

> **Попередня версія цього файлу містила дві серйозні помилки.** Я взяв `TOP 10` із `uniPartners` і побачив номери співробітників `1..10`, через що зробив неправильний висновок про "карти не використовуються". Також не міряв реальне використання `fDiscount` на позиціях.
>
> **Це виправлена версія** з фактичними цифрами по всіх ~16 859 документах і 274 партнерах.

---

## 1. Картки клієнтів — ВИКОРИСТОВУЮТЬСЯ (99 % покриття!)

```
fGroupID=2 "Клієнти":       217 партнерів, 216 з fCardNumber
fGroupID=3 "Постачальники":  57 партнерів,  56 з fCardNumber
                            ───────────────────────────────
Разом:                      274 партнерів, 272 з fCardNumber (99.3%)
```

Це **активна дисконтна картка**. Касир сканує штрих-код картки → Unipro знаходить партнера → застосовує його `uniContracts.fDiscount` + правила з `uniDiscount` + ручні знижки касира.

**Висновок для нас:**
- `mirror_partners.cardNumber` (уже синкаємо з `fCardNumber`) — **готове джерело істини** для номера картки клієнта.
- У Telegram WebApp показуємо клієнту **його існуючий номер картки** як barcode/QR — не генеруємо нові.
- Якщо клієнт не знайшов свою картку в Unipro — наш модуль може видавати **нові віртуальні номери** із власного діапазону.

## 2. Знижки — АКТИВНО ВИКОРИСТОВУЮТЬСЯ (17.7 % позицій із знижкою)

`uniDocGoods.fDiscount` (вже синкаємо як `MirrorDocumentItem.discount`):

| `fDiscount` | Позицій | % |
|------------:|--------:|---:|
| 0 (без знижки) | 24 131 | 82.3 % |
| < 5 %         | 2 921 | 10.0 % |
| 5–15 %        | 1 227 |  4.2 % |
| ≥ 15 %        | 1 034 |  3.5 % |
| **Усього з знижкою** | **5 182** | **17.7 %** |

Це **робоча знижкова система**, просто реалізована як:
- **Постійна знижка на партнера** через `uniContracts.fDiscount` (ще не синкаємо — треба додати!)
- **Ручна знижка касира** на конкретну позицію
- **Правила з `uniDiscount`** (2 правила у нашій інсталяції — швидше за все, валідатори + правило по групі)

## 3. Договори клієнтів (`uniContracts`) — НЕСИНКОВАНЕ ДЖЕРЕЛО

Ключові поля:

| Поле | Тип | Що означає |
|------|-----|------------|
| `fId` | bigint | ID договору |
| `fName` | nvarchar | "Основний договір" зазвичай |
| `fPartnerId` | int | → uniPartners |
| `fDiscount` | float | **Постійна % знижка для партнера** |
| `fPriceId`, `fPriceId2` | int | Індивідуальна цінова політика (опт/роздріб) |
| `fCurrId` | int | Валюта |
| `fType`, `fAccountType` | int | Тип договору |
| `fDefault` | int | Чи основний |

У топ-5 рядках `fDiscount = 0`, але механізм є. **Треба синкати** як `MirrorContract` (нова таблиця) або хоча б `mirror_partners.defaultDiscount`/`mirror_partners.priceListId` (грошово простіше).

## 5. Контакти партнерів (`uniPartners`)

6 полів телефону: `fTel`, `fTel1`..`fTel5`. 3 адреси: `fAdress`, `fAdress1`, `fAdress2`. 6 datetime: `fDate`..`fDate5`.

**Email-поля немає.**

Поточна заповненість у дзеркалі: 9/274 (3.3 %) мають телефон. Це або дані з паперових анкет не переносять, або касири не питають. **Перша engagement-кампанія = збір телефонів** (через бот, СМС після покупки, при виданні картки).

---

## 6. Групи партнерів (`uniPartnersGroups`)

```
1 root
2 Клієнти       ← 217 партнерів (216 з картою)
3 Постачальники ← 57 партнерів  (56 з картою)
```

Тобто **сегментація "клієнт vs постачальник" вже є в Unipro** через `fGroupID`. Наш RFM має фільтрувати тільки `fGroupID=2`.

## 7. Інші релевантні поля

- `uniDocuments.fPayCardId` — посилання на спосіб оплати/банк-картку.
- `uniDocuments.fCashAccountId` — на яку касу надійшла оплата (можна аналізувати безготівка vs готівка).
- `uniDocGoods.fCouponCode`, `fExStamp` — купони/штампи (вже синкаємо як `couponCode`, `exStamp`).
- `uniUsers.fCardNumber` — карти касирів (для входу в касовий софт).
- `uniGoods.fPriceOut1..fPriceOut10` — **10 рівнів цін** для різних категорій клієнтів (`uniContracts.fPriceId` обирає, який саме застосовується).

## Архітектурні висновки для Stage 4 (виправлені)

1. **Картки клієнтів — НЕ генеруємо нові, читаємо з Unipro.** `mirror_partners.cardNumber` є SoT. У Telegram WebApp показуємо barcode/QR з цим номером.
   - Виняток: партнерам без `cardNumber` (1.1 %) видаємо віртуальний номер з нашого діапазону.
2. **Знижка клієнта (%) — комбінована:**
   - **Постійна** — `uniContracts.fDiscount` (треба засинкати).
   - **Поточна** — наш RFM-розрахунок з кешбеком/бонусами.
   - У віджеті касира/Telegram-боті показуємо обидві.
3. **Бонуси (кешбек) — повністю наші.** Рахуємо по `MirrorDocument` (`docType IN (2,4)`), зберігаємо в `crm_bonus_account.balance`. Касир бачить баланс у нашому віджеті, при списанні робить виклик нашого API і вручну знижує суму в Unipro.
4. **Email + день народження — наша анкета** в Telegram WebApp (в Unipro немає взагалі або є тільки в дзеркалі `dates` JSON).
5. **Phone collection — критична передумова.** SMS-кампанії неможливі при 3.3 % покритті.
6. **RFM — тільки `fGroupID=2` "Клієнти"**, постачальників виключаємо.
7. **Sync extension — додати:**
   - `MirrorContract` (uniContracts) або колонки `defaultDiscount` + `priceListId` в `MirrorPartner`.
   - Нормалізація `phones` в E.164 (libphonenumber-js, UA).
   - Можливо синкати `uniPrices`/`uniPricesNames`, щоб у боті показувати "оптова ціна XX% дешевша за роздріб".
